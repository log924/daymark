# GoReader 今日阅读路径：排序与筛选规格（v0.1）

## 产品目标

GoReader 的首页不是新闻热榜，也不是按 RSS 到达顺序排列的收件箱。它应在保留所有订阅内容的前提下，给出一条有限、可解释、跨主题的个人阅读路径。

核心问题是：**这篇内容在今天是否仍值得占用一个阅读位置？**

## 已知数据事实（2026-07-18 本地快照）

- 496 篇文章来自 10 个来源；其中 cnBeta 241 篇（49%），The Verge 90 篇（18%）。
- 32 篇已有已读信号。低产来源 MilkShakism、Louiscard、Five Books 的已读率高于高产科技媒体；但样本尚小，不能据此自动重置来源权重。
- 当前库内存在相同 canonical URL 的重复文章；抓取时只按原始 URL 查询，尚未有数据库唯一约束或统一 URL 规范化。
- 日报刷新通过并发抓取累积新增文章，再截取前 80 篇交给模型。候选顺序会受 RSS 响应速度影响，不稳定。

这些事实说明：来源产量不能提升优先级；确定性去重和稳定候选排序应先于复杂模型。

## 不变原则

1. **全部保留**：不进入今日阅读路径不等于删除，文章仍可在全部动态中检索和打开。
2. **事件去重优先于排序**：同一事件只消耗一个首页位置，其他报道以“相关报道”折叠。
3. **来源多样性优先于来源产量**：高产媒体提供更多候选，但不会因此获得更多默认首页位置。
4. **多样性是软约束**：它降低重复来源／主题的边际价值，不能拦截有明显新增事实或足够重要性的内容。
5. **模型描述，代码决策**：模型输出结构化维度；最终分、阈值、惩罚及例外均由可调代码计算。
6. **所有入选项可解释**：每篇首页文章都能说明 `Why now`，包括主要正向信号和适用的多样性惩罚。
7. **促销内容不参与精选**：标题明显属于优惠、折扣、历史低价、优惠券或促销汇总的文章保留在全部动态，但不进入评分、阅读路径或日报。

## 建议流程

1. 抓取 RSS，保存原始文章。
2. 规范化 URL、标题、发布时间、语言与正文长度。
3. 执行确定性去重：canonical URL、规范化 URL、同源相同标题／时间窗口。
4. 为新文章创建候选记录；按发布时间倒序稳定排序，不能按抓取完成顺序截断。
5. 轻量模型标注（可异步）：主题、内容形态、信息密度、原创／一手性、与个人兴趣的贴合度、可读性，以及可用于聚类的简短事件描述。
6. 代码计算基础分并做事件聚类。
7. 通过“边际价值”逐篇构建今日阅读路径：来源或主题已经出现时施加软惩罚；足够重要的新内容可突破惩罚。
8. 对已入选内容生成中文摘要／翻译和日报；未入选内容不为此支付额外模型成本。

## 分数设计

所有维度归一化到 0–100。初版不引入外部热度指标。

```text
baseScore =
  0.26 × personalRelevance
  0.20 × sourceTrust
  0.18 × informationValue
  0.16 × novelty
  0.12 × freshness
  0.08 × explicitFeedback

marginalScore =
  baseScore
  - sourceExposurePenalty
  - topicExposurePenalty
  - clusterOverlapPenalty
```

- `personalRelevance`：主题、作者、来源和书单／已读内容的长期关联；早期数据不足时保持中性，避免以少量已读记录过拟合。
- `sourceTrust`：用户设定的来源信任与来源类型（原始资料、媒体、评论、聚合）。它是质量先验，不随当天发文量增加。
- `informationValue`：模型判断的具体性、深度、原创性和可行动性。
- `novelty`：相对近 7 天候选是否提供新的事实、角度或实用价值；同一事件的改写分数低。
- `freshness`：平滑时间衰减。常青内容不应被直接归零。
- `explicitFeedback`：收藏、读完、标记“多一点／少一点”等高置信信号；单纯曝光不是正反馈。

### 软约束与例外

- 同一来源或主题每增加一个已入选项，后续候选的惩罚递增。
- 同一事件簇的非主条通常不进入首页；主条选择优先一手性、信息完整度和个人相关性。
- 若文章的 `baseScore` 高于“重大内容阈值”，且它为事件簇带来新事实、独家信息或明确的实用价值，则可突破来源／主题惩罚。
- 例外必须在 `Why now` 中显式呈现，例如“虽然今日已选 2 篇 The Verge，但这是独家调查并带来新事实”。

阈值和权重应版本化保存，任何调整均可回测过去候选集。

## 数据与界面最小改动

初版需要新增文章处理状态与可审计结果，而不是替换现有原文：

- 文章：规范化 URL／标题、抓取时间、语言、去重主条 ID、主题和内容形态。
- 评分：维度分、最终基础分、边际分、评分版本、`Why now` 数据。
- 来源：用户信任权重、来源类型、可选主题标签。
- 事件簇：簇 ID、主条、相似文章关系。
- 反馈：收藏、读完、隐藏、`more`／`less` 的显式偏好事件。

界面保留“全部动态”，新增或强化“今日阅读路径”。每项显示来源、主题、发布时间、`Why now` 与“相关报道”。

## 实施顺序

1. **确定性基础**：URL 规范化、数据库唯一性、稳定的候选排序，修复当前同 URL 重复与并发顺序问题。
2. **可解释的无模型排序**：来源配置、时间衰减、软多样性惩罚、首页干跑结果与 `Why now`。
3. **用户反馈**：低摩擦的 `more`／`less` 与隐藏／收藏信号，并只用于可调权重。
4. **轻量 AI 标注**：替换当前“模型直接给个人分”的单维机制；先提供主题、类型、信息价值和事件描述。
5. **语义事件聚类**：在确定性去重稳定后，用 embedding 辅助识别跨语言或改写报道。
6. **日报重构**：日报只整理已经入选的候选，模型不再兼任整套筛选器。

## 成功标准

- 高产来源不会仅因数量占据首页，但高价值内容可越过软约束。
- 同 URL 内容不重复入库；同一事件在首页默认只有一个主条。
- 新增 RSS 的候选集合与排序不受抓取完成顺序影响。
- 每篇入选文章均能给出稳定、可读的推荐原因。
- 调整任一阈值或权重可重新计算历史结果，不需要改 Prompt。
